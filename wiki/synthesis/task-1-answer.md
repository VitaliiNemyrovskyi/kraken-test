# Task 1 — Автоматизована генерація SEO-сайтів

## Контекст

Завдання просить описати систему, яка автоматично робить SEO-сторінки під ключові запити: від отримання задачі до публікації готового сайту на Cloudflare Pages. Потім — як це масштабується до тисяч сторінок на місяць, які моделі та інструменти, скільки часу на побудову.

Перш ніж проектувати — потрібно прийняти один факт. У 2026 році Google прямо штрафує сайти, які мажуть AI-контентом без редакторського контролю (березневі core updates 2024 і 2026, грудневий 2025). Сайти що публікують сотні-тисячі AI-сторінок без edit-loop отримують падіння трафіку 50-87%. Але AI-контент **з** редакторським процесом — нормально проходить. Тому **архітектура цілеспрямовано будується навколо петлі редакторського контролю**, а не навколо "набили промпт — публікуй".

Друга деталь: це iGaming. Для голландського ринку (KSA — Kansspelautoriteit) є специфічні вимоги до реклами казино: заборона показувати знаменитостей і спортсменів, заборона націлювання на 18-24, обовʼязкове посилання на Cruks (реєстр самовиключення), вказання ліцензії. Жоден готовий вендор такі правила не знає — це окремий шар якого немає в комерційних продуктах типу SearchAtlas чи AirOps (див. [[task-1-prior-art]]).

## Як це працює — крок за кроком

### 1. Звідки беруться задачі

Підтримуємо **два способи** паралельно — і Google Sheets, і вебінтерфейс — щоб різні команди могли працювати як їм зручніше.

**Google Sheets** використовується SEO-командою для пакетного завантаження. У таблиці є колонки: ключове слово, ринок (geo), бренд, тип сторінки (review / comparison / FAQ-page), пріоритет, статус. Інтеграція через Google Sheets API: фоновий процес раз на 5 хвилин читає нові рядки з статусом `pending`, додає їх у чергу задач, змінює статус на `queued`. Готові сторінки потім записують назад URL опублікованого сайту і фінальний статус (`published` / `failed` / `under_review`) у ту саму таблицю.

**Вебінтерфейс** — для одноразових задач і ручного редагування. Менеджер відкриває дашборд, натискає "Нова сторінка", заповнює форму, бачить чергу і статуси в реальному часі. Це той самий React-фронтенд що в Task 2, тільки розширений.

Усе йде в спільну чергу — байдуже звідки задача прийшла. Черга реалізована як таблиця в PostgreSQL з полями: `id`, `source` (sheets / web), `payload` (JSON), `status`, `attempts`, `created_at`, `started_at`, `completed_at`. Воркери беруть задачі по черзі, оновлюють статус, у разі помилки роблять до 3 повторних спроб з експоненційною паузою.

### 2. Збір видачі Google

Для кожної задачі система робить запит до **DataForSEO Standard Queue**. Це API яке повертає топ-10 органічних результатів Google під заданий запит і geo, плюс People-Also-Ask питання і Related Searches.

Ціна: **$0.60 за тисячу запитів** (Standard queue, ~5 хвилин затримки). Це у 25 разів дешевше за SerpAPI ($15/1k) на тому самому обʼємі. Для тисячі нових сторінок на місяць — це 60 центів видатків на SERP-дані.

Якщо потрібна швидкість — той самий API має Priority queue ($1.20/1k, ~1 хв) і Live queue ($2.00/1k, ~6 секунд). Для бекграунд-генерації Standard queue зазвичай достатньо.

Що повертається і зберігається в базі: URL, домен, заголовок Google-сніпета, опис, позиція, плюс окремо PAA-питання і Related Searches. Це сирий матеріал для наступного кроку.

### 3. Сканування сторінок конкурентів

Для кожного з топ-10 URL воркер запускає Playwright Chromium і збирає зі сторінки конкурента:

- Заголовок H1
- Усі H2 і H3 — це найважливіше, бо з них формується структура нашої сторінки
- Перший абзац (intro) — щоб зрозуміти як конкуренти позиціонують тему
- Усі FAQ-секції (якщо є)
- Кількість слів у тексті — щоб орієнтуватися на середній обʼєм для цього запиту
- Зовнішні посилання — особливо посилання на офіційний сайт бренду чи на конкурентні бренди, з афіліатними мітками
- Schema-розмітку зі сторінки (`Article`, `FAQPage`, `Product`, `Review`)
- Наявність зображень, відео, таблиць
- Завантажуваність — час до першого корисного контенту

Сторінки які за Cloudflare bot-protection обходимо через **Bright Data Web Unlocker** ($4 за 1k запитів) або через резидентні проксі. Це не для всіх — тільки для тих що Playwright не може взяти власноруч. На реальному NL-ринку це приблизно 30-40% казинних сайтів.

Усе зібране йде в таблицю `competitor_scrapes`, привʼязане до задачі. Сирий HTML також зберігаємо в обʼєктному сховищі (R2 від Cloudflare) — на випадок якщо знадобиться повторно перепарсити з іншими правилами.

### 4. Формування структури контенту

Тут починається власне AI-частина. Перший виклик — побудувати **бриф сторінки**: яка має бути структура, які ключові теми покрити, які питання у FAQ, які сутності згадати.

Бриф будується **в один LLM-виклик** на основі зібраного матеріалу. Промпт виглядає приблизно так:

```
Ти будуєш SEO-структуру сторінки під запит "starcasino review"
для голландського ринку.

Ось топ-10 видачі Google і структура цих сторінок:
[список конкурентів з їхніми H1, H2, H3]

Ось People-Also-Ask: [питання]
Ось Related Searches: [фрази]

Завдання:
1. Запропонуй H1 (60-65 символів, природний, з ключовим словом)
2. 6-8 H2 секцій з підрозділами H3 — таких, що покривають теми
   які присутні мінімум у 4 з 10 конкурентів
3. 5-7 FAQ-питань — використовуй PAA, але переформулюй
4. Список сутностей які мають бути згадані: ліцензія KSA, конкретні
   ігрові провайдери, способи оплати характерні для NL (iDEAL, Bancontact)
5. Внутрішні посилання — які інші сторінки нашого сайту згадати

Обовʼязкові обмеження:
- Тон інформативний, не маркетинговий
- Заборона: знаменитості, спортсмени, "обовʼязково грайте"
- В інтро мають бути слова "18+", у футері — посилання на Cruks
```

Відповідь LLM — структурований JSON з усіма блоками. Цей JSON зберігається як `page_brief` для конкретної сторінки і стає вхідними даними для наступного кроку.

Для цього кроку використовуємо **Claude Opus 4.7 через OpenRouter** — це найвпевненіший в роботі з контекстом і структурою. Виклик на одну сторінку — приблизно $0.05-0.10.

### 5. Як AI пише сам текст

Маючи бриф — генеруємо **кожен блок окремо**, не всю сторінку одним викликом.

Для кожного H2-блоку йде окремий виклик до LLM з:
- Загальним контекстом сторінки (бриф)
- Заголовком блоку і підпунктами
- Виписаними цитатами з конкурентів які покривають цю тему
- Стилістичним гайдом бренду
- Списком заборонених фраз (KSA-обмеження)

Це дає дві переваги:
- **Якість.** Кожен блок отримує цілеспрямований промпт, не розмазаний на 2000 слів.
- **Селективна перегенерація.** Якщо один блок вийшов слабкий — переписуємо тільки його, не цілу статтю. Деталі цього механізму — окрема розмова, але по суті: кожен блок у базі знає звідки взявся (промпт, дані, модель), і може бути регенерований без впливу на сусідів.

Для більш масових і менш критичних сторінок (наприклад варіації під різні міста на тій самій темі) можна використовувати дешевшу модель — Sonnet 4.6 або GPT-4o-mini. Це робиться через **OpenRouter**, який дозволяє переключати моделі без зміни коду — просто інший рядок у конфігу. Економія: Opus 4.7 коштує ~$15 за мільйон вихідних токенів, Haiku 4.5 — ~$1.25. На сторінках де якість критична — Opus. На масових варіаціях — Haiku.

Вартість на одну повну сторінку (2000 слів, 6-8 блоків) у середньому: $0.30-0.80 залежно від моделі.

### 6. Контроль якості SEO

Згенеровані блоки проходять **два рівні перевірок** перш ніж піти на публікацію.

**Перший рівень — програмні правила.** Це швидкі детерміновані перевірки:
- Довжина блоку в межах очікуваного (не порожньо, не зашкалено)
- Ключове слово згадано (але не передозовано — keyword density 0.5-2%)
- Усі заплановані сутності зі брифу присутні
- Внутрішні посилання вставлені
- Schema-розмітка валідна (через `schema.org` validator)
- Нема дубльованих абзаців з інших сторінок нашого сайту (lexical similarity check)
- Compliance-перевірки KSA: є дисклеймер "18+", є посилання на Cruks, нема заборонених фраз (список регулярних виразів)

Якщо щось не пройшло — задача повертається на той блок який провалився, з відповідним коментарем у промпт ("FAQ не містить питання про ліцензію KSA, додай").

**Другий рівень — LLM-judge.** Окремий LLM-виклик (зазвичай швидша/дешевша модель, типу Sonnet 4.6) дивиться на готовий блок і відповідає на питання типу:
- Чи звучить це природно або як штамп?
- Чи є фактичні твердження що можуть бути неправдиві?
- Чи тон відповідає бренд-гайду?
- Чи відрізняється від того що пишуть конкуренти, чи це той самий буклет?

LLM-judge дає оцінку 1-5 і коментар. Блоки з оцінкою менше 3 автоматично йдуть на перегенерацію (один раз) або помічаються як `under_review` — їх потрапляє в чергу до людини-редактора.

Це і є той самий "editorial oversight loop" про який Google говорить у своїй документації. Він автоматизований, але існує.

### 7. SEO-розмітка HTML

Збираємо готову сторінку через **Astro 5** — генератор статичних HTML-сайтів. Astro обрано бо:
- Видає чистий HTML без зайвого JavaScript (Cloudflare нагороджує швидкість)
- Має готовий адаптер для Cloudflare Pages
- Підтримує partial rebuilds — це важливо для селективної перегенерації

Кожна сторінка має шаблон з трьома типами елементів: статичні (header, footer, навігація), напівдинамічні (контент сторінки — підставляється з блоків у базі), мета-теги (генеруються автоматично за брифом).

**Schema.org розмітка** додається автоматично через JSON-LD:
- `Organization` schema на всіх сторінках (інформація про сам бренд)
- `Article` або `Review` schema на основній сторінці (заголовок, автор, дата, рейтинг)
- `FAQPage` schema на блоці FAQ (питання-відповіді у структурованому форматі)
- `BreadcrumbList` для навігаційних крихт
- `Product` schema якщо це сторінка з оглядом конкретної гри/казино

**Мета-теги** які генеруємо: `<title>` (з брифу, 50-60 символів), `<meta description>` (з брифу, 150-160 символів), Open Graph (для соцмереж), Twitter Cards, canonical URL, hreflang якщо є переклади, robots.

Це все працює автоматично — редактор не торкається мета-тегів вручну, бо вони генеруються з даних. Якщо редактор змінив заголовок блоку — мета-теги перегенеровуються разом.

### 8. Публікація на Cloudflare Pages

Astro збирає всі сторінки в папку зі статичними HTML-файлами. Потім система викликає **Cloudflare Pages REST API** з командою "оновити цю папку".

Cloudflare приймає файли, кладе їх у свою глобальну мережу (CDN) — за кілька секунд оновлення доходить до всіх локацій у світі. Користувач у Нідерландах побачить нову версію сторінки одразу.

Декілька важливих моментів:
- **Атомарність публікації.** Cloudflare публікує всю партію за раз. Якщо в середині щось зламається — старі версії продовжують працювати, нова не зʼявиться частково.
- **Превʼю перед публікацією.** Cloudflare Pages дозволяє створити "preview deployment" з тимчасовим URL-ом — це використовуємо для перегляду перед фінальним розгортанням.
- **Відкат.** Якщо опублікували погану версію — Cloudflare зберігає історію всіх deployments. Один клік (або API-виклик) повертає попередню.
- **Безкоштовно.** Cloudflare Pages дозволяє безкоштовно до 500 build-ів на місяць і необмежений трафік. Для тисячі-двох сторінок це буде в безкоштовному рівні. Платний план починається коли треба більше білдів.

### 9. Оновлення статусу і подальше редагування

Коли сторінка опублікована — система робить декілька речей паралельно:

**У Google Sheets** — оновлюється рядок: статус `published`, URL опублікованої сторінки, дата публікації, можливо метрики (наприклад очікувана позиція якщо вже є rank-tracker налаштований).

**У базі даних** — записується подія в `publication_log`: яка сторінка, коли, які блоки змінилися порівняно з попередньою версією, хто ініціював.

**У вебінтерфейсі** — статус у дашборді змінюється в реальному часі через WebSocket-події.

**Подальше редагування** працює двома способами:

**Перший — селективна перегенерація.** Редактор у дашборді бачить сторінку, поділену на блоки. Біля кожного блоку є кнопка "Перегенерувати". Натискаючи, можна додати коментар ("FAQ слабке, питання банальні"). Система бере вихідний бриф цього блоку, додає коментар і генерує заново — за 2-3 секунди. Редактор бачить порівняння старого і нового, приймає або відхиляє. Тільки той блок змінюється в базі, сторінка перезбирається, публікується.

**Другий — точкове редагування виділеного тексту.** Редактор виділяє довільний шматок тексту, тисне "Переписати виділене". Система обгортає виділене як новий мікроблок, відправляє в LLM з контекстом сусідніх речень і інструкцією від редактора, повертає варіант. Деталі цього механізму — окрема розмова, але технічно це працює.

**Автоматична перегенерація** — окремий процес фонового моніторингу. Якщо сторінка втратила позиції в Google, або має високий показник відмов, або просто не оновлювалася понад 6 місяців — система автоматично ставить її в чергу на оновлення. Дані конкурентів збираються заново, бриф перегенерується, блоки які найбільше відрізняються від актуальної видачі — переписуються.

## Як це масштабується до тисяч сторінок на місяць

Базова цифра — 1000 нових сторінок на місяць. Це приблизно 50 сторінок на робочий день, або по одній кожні 10-15 хвилин у робочий час.

### Обмеження які реально зустрінемо

**Перше — обмеження зовнішніх API.**
- DataForSEO Standard queue: затримка ~5 хвилин на запит, але паралельність необмежена в рамках кошторису. На 1000 запитів — $0.60.
- OpenRouter / Anthropic: rate limit на ключ. Для Claude Opus 4.7 — приблизно 50 запитів на хвилину на стандартному тарифі. Якщо потрібно більше — або кілька ключів з ротацією, або платний tier (enterprise pricing).
- Cloudflare Pages: 500 безкоштовних build-ів на місяць. Платний план починається з $20/міс за 5000 build-ів.

**Друге — Playwright дорогий за ресурсами.** Один Chromium-екземпляр зʼїдає ~500MB RAM. Якщо хочемо паралельно сканувати 20 сторінок — це 10GB RAM тільки на скрейпінг. Рішення — окремі воркер-контейнери для скрейпінгу, горизонтальне масштабування, спільне зберігання сирого HTML у R2.

**Третє — якість при масштабі.** Чим більше сторінок ми згенеруємо за день, тим більший ризик що вони почнуть бути схожими одна на одну. Це Google вмить помітить (near-duplicate detection). Контроль — програмна перевірка лексичної подібності між згенерованими блоками, бажано на рівні sentence embeddings (через CLIP або sentence-transformers).

**Четверте — iGaming compliance не масштабується автоматично.** KSA правила змінюються, додаються нові заборони. Список заборонених фраз треба регулярно оновлювати. Це робота для людини — раз на місяць подивитися останні рішення регулятора і поновити правила.

### Як обходимо

**Архітектура — черга задач + горизонтально масштабовані воркери.** Воркери не мають стану, можна додавати/прибирати залежно від навантаження. PostgreSQL як черга для простоти (краще ніж Redis для нашого обʼєму), або BullMQ якщо стане більше 10 тисяч задач на день.

**Кешування скрейпів конкурентів.** Один і той самий сайт-конкурент може бути в топ-10 для багатьох наших запитів. Скрейпимо раз на 7 днів, потім результати беремо з кешу.

**Партіонування генерації.** AI-виклики групуємо в батчі по 5-10 паралельних. Якщо один з ключів OpenRouter уперся в rate limit — наступний батч іде через інший ключ.

**Моніторинг якості.** Окремий dashboard показує метрики: середній SEO-score, % сторінок які пройшли automated quality checks з першого разу, % які пішли на ручний огляд, середній час від задачі до публікації. Якщо середній SEO-score падає — алерт у Slack.

### Орієнтовна вартість при 1000 сторінок/міс

- DataForSEO SERP: $0.60
- Bright Data для важких казинних сайтів (~30% від 10 конкурентів × 1000 = 3000 викликів): $12
- LLM-виклики (Opus для брифу, Haiku/Sonnet для тіла, Sonnet для judge): ~$300-500
- Cloudflare Pages: $20 (Pro tier)
- Compute (2-3 невеликі VPS / контейнери): $50-100
- WHOIS / supporting APIs: ~$10

**Разом: $400-650 на місяць** — собівартість самої системи. Це проти ~$2000/міс за готову платформу типу AirOps Pro, яка все одно не вміє iGaming і KSA-правила.

## Стек і моделі

**Бекенд:** Node.js 20, TypeScript, Fastify, Playwright (Chromium), PostgreSQL (для черги задач і метаданих), Cloudflare R2 (для сирих скрейпів і обʼємного сховища).

**AI-шар:** OpenRouter як єдиний шлюз до різних моделей.
- Claude Opus 4.7 — на найважливіші виклики (генерація брифу, складні блоки)
- Claude Sonnet 4.6 або GPT-4o — для генерації тіла масових сторінок
- Claude Haiku 4.5 або GPT-4o-mini — для LLM-judge перевірок
- text-embedding-3-small (OpenAI) — для перевірок подібності між згенерованими блоками

**SERP-шар:** DataForSEO як основний, Bright Data Web Unlocker як fallback для bot-protected сайтів.

**Генератор сайтів:** Astro 5 з адаптером для Cloudflare Pages. Tailwind для стилів. Mermaid для діаграм усередині контентних сторінок (якщо потрібно).

**Хостинг:** Cloudflare Pages для самих SEO-сторінок. Окремий VPS (Contabo або Hetzner) для воркерів-генераторів і дашборду — щоб не платити за Cloudflare Workers compute на довгих AI-викликах.

**Моніторинг:** Sentry для помилок, простий dashboard у самому додатку для бізнес-метрик. Алерти в Slack.

## План впровадження

### Фаза 1 — MVP (4 тижні)

Мета — довести що пайплайн працює end-to-end на одній сторінці з ручним тригером.

- **Тиждень 1.** Базова інфраструктура: PostgreSQL з чергою задач, Fastify-API для прийому задач, Astro-проект з одним шаблоном сторінки, Cloudflare Pages з ручним деплоєм.
- **Тиждень 2.** Інтеграція DataForSEO для SERP, Playwright для скрейпу конкурентів. Зберігання даних у базу.
- **Тиждень 3.** Генерація брифу і одного блоку через Claude. Збірка HTML через Astro. Програмні quality checks (без LLM-judge поки що).
- **Тиждень 4.** Деплой через CF Pages API. Перша публікація однієї повної згенерованої сторінки. Дашборд (мінімальний — статус черги).

**Результат:** одна реально опублікована сторінка яку показуєш стейкхолдеру. Зрозуміло чи концепція робоча.

### Фаза 2 — Production-ready (8 тижнів)

Мета — стабільна система яка може робити 100-200 сторінок на тиждень з нормальним рівнем якості.

- **Тижні 5-6.** Селективна перегенерація на рівні блоків. Інтерфейс редагування в дашборді. Запис історії версій кожного блоку.
- **Тиждень 7.** LLM-judge другої лінії перевірки. Інтеграція з Google Sheets (читання задач + запис статусу).
- **Тиждень 8.** KSA compliance шар: список заборонених фраз, перевірка на наявність обовʼязкових елементів (18+, Cruks, ліцензія).
- **Тиждень 9.** Кешування скрейпів конкурентів. Bright Data fallback для важких сайтів.
- **Тиждень 10.** Schema.org розмітка для всіх типів сторінок. Внутрішня перелінковка. Sitemap-генерація.
- **Тиждень 11.** Моніторинг якості (середній SEO-score, % переходів на ручну перевірку), алерти.
- **Тиждень 12.** Тестування на 100 сторінках, фіксяг знайдених проблем, документація.

**Результат:** система яку можна показувати клієнту як готовий продукт.

### Фаза 3 — Scale-out (12+ тижнів далі)

Мета — масштабування до 1000+ сторінок на місяць, додаткові ринки і мови.

- Точкове редагування виділеного тексту (механізм мікроблоків)
- Автоматична перегенерація на основі performance-даних (інтеграція з Google Search Console API)
- A/B-тестування варіантів блоків
- Підтримка інших ринків (Німеччина, Італія, Португалія) — окремі compliance-шари для кожного регулятора
- Багатомовність — той самий пайплайн на німецькій / італійській / португальській
- Розширений API для зовнішніх клієнтів (інших команд всередині Kraken)
- Аналітика рентабельності кожної сторінки (вартість генерації проти заробленого трафіку)

## Висновок

Завдання вирішується. Основна частина пайплайну — це збірка з зрілих компонентів які вже існують (DataForSEO, Playwright, Claude через OpenRouter, Astro, Cloudflare Pages). Скласти все це в один робочий потік — приблизно 12 тижнів роботи однієї людини, або 6-8 тижнів вдвох.

Те що відрізняє цю систему від готових платформ типу SearchAtlas чи AirOps — три речі: (1) **iGaming + KSA compliance** який жоден вендор не покриває з коробки, (2) **редакторський контроль на рівні блоків та виділених шматків** який вони не пропонують, (3) **прямий доступ до різних AI-моделей через OpenRouter** замість блокування на один LLM з націнкою у 5-10 разів.

Перші дві речі — це те що робить систему **реально працездатною** для нашого випадку. Третя — це те що робить її **рентабельною** на масштабі: $400-650/міс собівартості проти $2000+/міс за готову платформу яка все одно не вміє те що нам треба.

Найбільший ризик — **не технічний, а редакторський**. Google у 2026 році активно карає чистий AI-bulk. Виграти можна тільки якщо реально є шар редакторського контролю, навіть автоматизованого. Без нього на третьому місяці система почне втрачати позиції, не набираючи їх.
